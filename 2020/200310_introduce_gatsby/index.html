Gatsbyの導入

GatsbyReact

March 10, 2020

このブログはもともと静的サイトジェネレータのHugoを用いて生成したページをGitHub Pagesを利用して公開していたのですが、 2020年になって久しぶりに更新しようかなと思ったところ、すっかりHugoの使い方を忘れてしまっていました。
そういう理由に加え、また仕事柄Reactを使用するようになったこともあり、せっかくなら新しい静的サイトジェネレータに移行しようかなと思った次第です。

Gatsbyとは

GatsbyはReactをベースにした静的サイトジェネレータです。
静的サイトジェネレータという名の通り、事前にビルドを行うことによりhtmlファイルとjavascriptファイルを生成します。
通常のReactも同じようにビルドしているじゃないかと思ったのですが、Reactではよくバンドルされたjavascriptファイル(とcssファイル)が生成されるのに対し、Gatsbyは基本的にhtmlファイルを生成します。
ここが静的サイトジェネレータと言われる所以だと思います。
一方Reactをベースにしているので、記述方法はjsxですし、なんならjavascriptでロジックを書くことも可能です。

じゃあGatsbyの利点って?

ReactによるSPA(Single Page Application)に比べて何がいいの?という疑問に対しては、SPAでごりごりロジックを入れたページを作りたいのでなければ圧倒的に速度が出るというのが静的サイトジェネレータであるGatsbyの利点かなと思います。
(このブログもやろうと思えばReactとReact-Routerを使ってSPAで作れなくもないですが…)
また、Gatsbyが用意しているLinkコンポーネントを利用することで、ページ内にリンクがあればプリフェッチしてくれるので高速にリンク先に遷移できることもメリットです。

HugoからGatsbyへの移行

スターターセットを用いたGatsbyの導入

本題に入ります。
Gatsbyの導入方法はいろいろありますが、自分はもともとHugoを使っていた際に記事をMarkdownファイルとして書いていたこともあり、Markdownファイルを殆どそのまま流用できるgatsby-starter-blogというスターターセットを利用して導入しました。
スターターセットとは、言わば予め必要そうなモジュールの導入や設定が施されたテンプレートのようなものです。 上記のgatsby-starter-blogはその名の通り、ブログ向けのスターターセットになります。

まずターミナルで以下のコマンドを打ちます。(hogehogeはGatsbyにより構築されるファイル群が含まれるフォルダ名となります) 

gatsby new hogehoge https://github.com/gatsbyjs/gatsby-starter-blog

次にhogehogeに移動して、gatsby developを実行します。
そしてlocalhost:8000に移動すると、初期ページが表示されるはずです。

Gatsbyのフォルダ構造

gatsby-starter-blogにより導入したGatsbyのフォルダ内の構造について見ていきます。

contentフォルダ

ここには、サイト全体のリソースファイル(画像とか)があるassetsフォルダと、Markdownで書かれたブログの記事が配置されるblogフォルダがあります。
blogフォルダ以下にMarkdownで記事を書き保存することで、ビルド時にgatsby-source-filesystemプラグインによりデータベース化され、それをもとにhtmlファイルが生成されます。
因みにblogフォルダ以下のツリー構造はそのままサイトのURLに反映されます。
例えばこの記事のURLはhttps://tonooo71.github.io/2020/200314_introduce_gatsby/ですが、記事自体はblog/2020/200314_introduce_gatsby/フォルダ内にあるindex.mdとgatsby_icon.pngから構成されています。
どのフォルダをコンテンツのソースとするかはgatsby-config.jsに書かれてあるので、他にコンテンツソースフォルダを作りたいとなったときはここを編集する必要があります。
もともとサンプルの記事があるので、Hugoで書いていたMarkdownファイルもそれを参考に少し書き換えました。
とは言っても最初に書くメタデータの書式以外は同じMarkdownファイルなので、殆どそのまま利用できるのではないかと思います。

publicフォルダ

このフォルダはyarn buildでビルドした際に生成されるサイトのソースが保存されます。
このフォルダ内のものをサーバーにアップロードする必要があります。

srcフォルダ

componentsフォルダには、ウェブサイト共通のページの外観(layout.js)や記事一覧ページなどに表示されるブログ記事へのリンク要素(page.js)、著者を表示する要素(bio.js)などが書かれたReactコンポーネントが保存されています。
pagesフォルダには、トップページの外観(index.js)や404ページの外観(404.js)が書かれたReactコンポーネントがあります。
templatesフォルダには、ブログ記事ページのテンプレートとなるblog-post.jsがあり、ここにデータベースから記事の内容を流し込むことでHTML化された記事ページが生成されます。
いずれもJSXファイルなので、Reactに慣れている人であれば理解しやすいとは思います。

その他

rootディレクトリ内のgatsby-browser.jsやgatsby-config.js、gatsby-node.jsはプラグインの導入やページの生成の際に変更するものです。
後々機能を追加していったりする際には編集必須です。(未だに全部はよくわかっていないのですが)

スクリプト

Hugoはデフォルトでデプロイのためのスクリプトが用意されていたように思いますが、Gatsbyには予め用意されていないので作成してみました。

以下は、ビルドを行いpublicフォルダ内のファイルをgithubioフォルダにコピーし、リモートリポジトリoriginに対しpushするスクリプトです。 

#!/bin/bash

yarn clean
yarn build

git add .

msg="rebuilding site `date`"
if [ $# -eq 1 ]
  then msg="$1"
fi
git commit -m "$msg"

git push origin master

cd ../githubio

git pull origin master
cp -r ../blog/public/* ./
git add .
git commit -m "$msg"
git push origin master

他にも、新規記事を作成するスクリプトなんかも作っておけば便利かもしれません。 

import datetime
import os
import sys


def new_blog(title):
    now = datetime.datetime.now()
    year = f"{now.year}"
    month = f"{now.month}".zfill(2)
    day = f"{now.day}".zfill(2)
    os.mkdir(f"./content/blog/{year}/{year[2:]}{month}{day}_{title}")
    with open(f"./content/blog/{year}/{year[2:]}{month}{day}_{title}/index.md", "w") as lines:
        lines.write("---\n")
        lines.write("title: \n")
        lines.write(f"date: {now.isoformat()}\n")
        lines.write("description: \n")
        lines.write("---\n")
        lines.write("\n")

if __name__ == "__main__":
    new_blog(sys.argv[1])
